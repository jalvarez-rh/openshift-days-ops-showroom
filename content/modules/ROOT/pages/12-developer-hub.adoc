= Platform Engineering - Red Hat Developer Hub

**Duration:** 30 minutes +
**Format:** Guided exploration + Hands-on

== The Problem

How many times a week does someone message you: _"Is my app running?"_ or _"Can you create me a namespace?"_

Red Hat Developer Hub (RHDH) is a developer portal based on Backstage. Your platform engineering team has deployed it for this organization, and it's already in use by multiple teams. As an ops admin, you need to understand what it does, what developers see when they use it, and how the pieces fit together so you can support it.

RHDH gives your organization:

* **Software Catalog** - a single place where every service, API, and library is registered. Developers find what exists, who owns it, and what state it's in - without asking you.
* **Software Templates** - self-service workflows that provision namespaces and resources using patterns the platform team defines. Developers get what they need without filing a ticket.
* **Plugins** - connect the catalog to live systems (Kubernetes, ArgoCD, Tekton, etc.) so developers see real-time status without needing cluster access.

In this module you'll explore a live Developer Hub that multiple teams are already using, see what developers see, watch a self-service template in action, and wire up a new service - the kind of task that lands on your desk when a team onboards.

== Understand How It Works

Before you start making changes, take a minute to understand the moving parts. When something breaks with RHDH, this is what you need to know.

=== Step 4: Check the RHDH Deployment

Verify the RHDH pods are running:

[source,bash,role="execute"]
----
oc get pods -n backstage
----

You should see:

----
NAME                                       READY   STATUS
backstage-developer-hub-xxxxx              1/1     Running
backstage-psql-developer-hub-0             1/1     Running
----

The main pod runs the Developer Hub application. The PostgreSQL pod stores the catalog data, user sessions, and template task history.

=== Step 5: Explore the Operator and CR

RHDH is managed by an Operator. The Operator watches a `Backstage` Custom Resource and reconciles the deployment whenever the CR changes.

View the Backstage CR:

[source,bash,role="execute"]
----
oc get backstage -n backstage
----

Look at how it's configured:

[source,bash,role="execute"]
----
oc get backstage developer-hub -n backstage -o yaml | grep -A5 "dynamicPlugins\|extraEnvs\|extraFiles\|appConfig"
----

This shows you the key configuration points:

* **dynamicPluginsConfigMapName** - which ConfigMap controls which plugins are enabled
* **extraEnvs** - environment variables and secrets injected into the pod (like the Kubernetes service account token)
* **extraFiles** - ConfigMaps mounted into the pod (like the catalog entries)
* **appConfig** - the RHDH application configuration

=== Step 6: Check the Plugin Configuration

See which dynamic plugins are enabled:

[source,bash,role="execute"]
----
oc get configmap dynamic-plugins-rhdh -n backstage -o jsonpath='{.data.dynamic-plugins\.yaml}' | grep -A1 "package:"
----

You should see the Kubernetes backend and frontend plugins, the Topology plugin, and the scaffolder modules that power the software templates. These are bundled in the RHDH image - the ConfigMap just switches them on or off.

=== Step 7: Understand the Label Contract

The Kubernetes plugin discovers pods by matching labels. This is the contract between the catalog and the cluster:

1. The catalog entry has an annotation: `backstage.io/kubernetes-id: weather-app`
2. The deployments have a matching label: `backstage.io/kubernetes-id: weather-app`
3. The plugin finds all pods with that label and displays them

Verify this on the weather app:

[source,bash,role="execute"]
----
oc get deployments -n ops-track-demo -o custom-columns='NAME:.metadata.name,BACKSTAGE-ID:.spec.template.metadata.labels.backstage\.io/kubernetes-id'
----

All 5 deployments should show `weather-app` as their backstage ID. If any deployment is missing this label, it won't appear in the Kubernetes tab. You'll see this in action during the troubleshooting section.

== Hands-On: Onboard a Service

The `order-tracker` service is registered in the catalog and now has a namespace, but it's not deployed yet. The inventory team has asked you to deploy their application. This is a common ops task - the catalog entry exists, the namespace exists, and you need to wire up the workloads.

=== Step 10: Deploy the Application

Deploy two microservices for the order tracking system:

[source,bash,role="execute"]
----
oc create deployment order-tracker-api --image=registry.access.redhat.com/ubi9/httpd-24 -n order-tracker
oc create deployment order-tracker-worker --image=registry.access.redhat.com/ubi9/httpd-24 -n order-tracker
----

Wait for the pods to start:

[source,bash,role="execute"]
----
oc rollout status deployment/order-tracker-api -n order-tracker --timeout=120s
oc rollout status deployment/order-tracker-worker -n order-tracker --timeout=120s
----

=== Step 11: Wire Up RHDH Discovery

The deployments are running, but if you check the **order-tracker** Kubernetes tab in Developer Hub right now, you still won't see any pods. The plugin doesn't know these deployments belong to this catalog entry.

Remember the label contract from Step 7? You need to add the `backstage.io/kubernetes-id` label to each deployment's pod template:

[source,bash,role="execute"]
----
for dep in order-tracker-api order-tracker-worker; do
  oc patch deployment $dep -n order-tracker --type=json \
    -p '[{"op":"add","path":"/spec/template/metadata/labels/backstage.io~1kubernetes-id","value":"order-tracker"}]'
done
----

Verify the labels are set:

[source,bash,role="execute"]
----
oc get deployments -n order-tracker -o custom-columns='NAME:.metadata.name,BACKSTAGE-ID:.spec.template.metadata.labels.backstage\.io/kubernetes-id'
----

=== Step 12: See It in Developer Hub

Go back to the **Developer Hub** tab and click on **order-tracker** in the catalog. Click the **Kubernetes** tab.

You should now see 2 pods running - `order-tracker-api` and `order-tracker-worker`.

image::rhdh-order-tracker-2pods.png[order-tracker now showing 2 live pods]

The inventory team's service went from a catalog placeholder to a live, visible deployment. They can now check the status themselves without messaging you. You did this by deploying the workloads and adding one label.

== Troubleshoot: A Service Disappeared

Here's a scenario you'll encounter in production: a developer reports that one of their services has disappeared from Developer Hub.

=== Step 13: Break Something

Let's simulate the problem. Remove the backstage label from one of the payments team's deployments:

[source,bash,role="execute"]
----
oc patch deployment payments-gateway -n payments-team --type=json \
  -p '[{"op":"remove","path":"/spec/template/metadata/labels/backstage.io~1kubernetes-id"}]'
----

=== Step 14: See the Impact

Go to the **Developer Hub** tab and click on **payments-platform** in the catalog. Click the **Kubernetes** tab.

You should now see only **2 pods** instead of 3. The `payments-gateway` deployment is still running - but it's invisible to Developer Hub.

image::rhdh-payments-broken-2pods.png[payments-platform showing only 2 pods after label removal]

This is the ticket you'd get: _"Our gateway service disappeared from Developer Hub but it's still running. Can you check?"_

=== Step 15: Diagnose and Fix

Check what's deployed versus what Developer Hub can see. Compare the deployments and their labels:

[source,bash,role="execute"]
----
oc get deployments -n payments-team -o custom-columns='NAME:.metadata.name,BACKSTAGE-ID:.spec.template.metadata.labels.backstage\.io/kubernetes-id'
----

You should see that `payments-gateway` has `<none>` for its backstage ID while the other two show `payments-platform`. The label is missing.

Fix it:

[source,bash,role="execute"]
----
oc patch deployment payments-gateway -n payments-team --type=json \
  -p '[{"op":"add","path":"/spec/template/metadata/labels/backstage.io~1kubernetes-id","value":"payments-platform"}]'
----

Go back to Developer Hub and check the **payments-platform** Kubernetes tab again. All 3 pods should be visible.

image::rhdh-payments-fixed-3pods.png[payments-platform restored to 3 pods after fix]

The fix was one label. But knowing _where_ to look - understanding the label contract between the catalog and the cluster - is what separates a quick fix from an hour of guessing.

== Going Further

This module showed you RHDH from an ops perspective - understanding the tool your developers use every day. A full platform engineering implementation adds:

* **SSO Integration** - connect to your identity provider so developers log in with their existing credentials
* **RBAC Plugin** - control who can register components, create from templates, or view specific services
* **CI/CD Plugins** - Tekton pipeline runs and ArgoCD sync status visible directly in the catalog
* **TechDocs** - documentation lives alongside the code and renders in the portal
* **More Software Templates** - golden paths for deploying applications, creating pipelines, and provisioning infrastructure
