= Secrets Management with HashiCorp Vault

== Module Overview

**Duration:** 25 minutes +
**Format:** Hands-on lab +
**Audience:** Platform Engineers, Security Engineers, IT Operations

== Learning Objectives

By the end of this module, you will:

* Deploy HashiCorp Vault on OpenShift using the official Helm chart
* Access the Vault UI and confirm the server is running in your cluster
* Understand how Vault fits into secrets management for Kubernetes workloads
* Store an application secret in Vault and deliver it to a pod with the Vault Agent Injector

== About HashiCorp Vault on Kubernetes

HashiCorp Vault stores and brokers access to secrets, certificates, and encryption keys. The official https://github.com/hashicorp/vault-helm[Vault Helm chart] installs Vault on Kubernetes or OpenShift and can optionally deploy the Vault Agent Injector for pod secret injection.

Workshop prerequisites match the upstream chart:

* Helm 3.6+ (the install script downloads Helm to `~/.local/bin` if it is not already installed)
* OpenShift / Kubernetes 1.29+

Reference: https://developer.hashicorp.com/vault/docs/platform/k8s/helm[Vault Helm documentation]

== Deploy Vault

The cluster provisioning script installs Vault into the `vault` namespace. If you are starting the lab without that step, run the installer from the bastion.

=== Option A — Setup script (recommended)

Run the installer from the `setup-scripts` directory in this workshop repo (on the bastion). The script installs Helm automatically if it is not already in your `PATH`.

[source,bash,subs="+macros,+attributes",role=execute]
----
cd setup-scripts
./install-vault.sh
----

If the repo is not on the bastion yet, clone it first:

[source,bash,subs="+macros,+attributes",role=execute]
----
git clone https://github.com/mfosterrox/openshift-days-ops-showroom.git
cd openshift-days-ops-showroom/setup-scripts
./install-vault.sh
----

The script:

* Adds the HashiCorp Helm repo (`https://helm.releases.hashicorp.com`)
* Installs chart `hashicorp/vault` into namespace `vault`
* Enables OpenShift integration (`global.openshift: true`)
* Uses **dev mode** for the lab (auto-initialized, auto-unsealed; **not for production**)
* Creates an OpenShift **Route** at `vault.apps.<cluster-domain>` (matches the cluster apps domain — not `chart-example.local`)
* Requests a **cert-manager** route certificate (when the operator is installed) so the TLS name matches the Route host
* Enables the **Vault Agent Injector** for Kubernetes secret injection patterns

Values file: `setup-scripts/vault-values-openshift-lab.yaml`

[NOTE]
====
If the route host is `chart-example.local` or pods show `ErrImagePull`, re-run the installer (it upgrades the release and fixes the route):

[source,bash]
----
cd setup-scripts
./install-vault.sh
----
====

=== Option B — Helm commands (manual)

[source,bash,subs="+macros,+attributes",role=execute]
----
helm repo add hashicorp https://helm.releases.hashicorp.com
helm repo update
oc create namespace vault --dry-run=client -o yaml | oc apply -f -
cd setup-scripts
./install-vault.sh
----

Run the commands above from the root of the `openshift-days-ops-showroom` clone.

[IMPORTANT]
====
**Dev mode** is enabled for this workshop so you can focus on Vault features without unseal keys or persistent storage setup. All data is lost when the Vault pod restarts. Do not use dev mode in production. For production on OpenShift, see https://developer.hashicorp.com/vault/docs/platform/k8s/helm/openshift[Vault on OpenShift].
====

== Access the Vault UI

The Route hostname must match the cluster apps domain, for example `vault.apps.cluster-nxxq9.dynamic2.redhatworkshops.io` — not `vault.apps.apps....` (double `apps`) and not `chart-example.local`.

When cert-manager is available, the install script requests a certificate for that exact hostname (`ClusterIssuer` `vault-lab-issuer`, self-signed for the lab). That removes `ERR_CERT_COMMON_NAME_INVALID`; your browser may still warn until you trust the lab CA or choose the exception.

. Open the *OpenShift Console* tab in the workshop panel.
. Go to *Networking* → *Routes*, project `vault`.
. Open the route named `vault` and confirm the host under *Location*.
. Log in with the dev root token: `root`

Or open the URL from the bastion:

[source,bash,subs="+macros,+attributes",role=execute]
----
echo "https://$(oc get route vault -n vault -o jsonpath='{.spec.host}')/"
----

== Build an application with Vault access

This workflow uses the **Vault Agent Injector** (installed with the Helm chart) to mount an application secret into a pod at runtime. The app never embeds the secret in its image or in an ordinary Kubernetes `Secret` object you manage by hand.

The pattern follows HashiCorp's Kubernetes sidecar injection guide; commands are adapted for this workshop's dev-mode Vault in the `vault` namespace.

=== Overview

. Store a secret in Vault (KV secrets engine v2).
. Enable the **Kubernetes auth method** so Vault trusts OpenShift service account tokens.
. Create a Vault **policy** and **role** for the application service account.
. Deploy a sample application whose pod annotations request secret injection.
. Confirm the secret file appears in the pod and the application can read it.

=== Step 1 — Configure the Vault CLI on the bastion

. Set `VAULT_ADDR` to your OpenShift route (or port-forward) and the dev root token.
+
[source,bash,subs="+macros,+attributes",role=execute]
----
export VAULT_ADDR="$(oc get route vault -n vault -o jsonpath='https://{.spec.host}')"
export VAULT_TOKEN=root
export VAULT_SKIP_VERIFY=true
vault status
----

[NOTE]
====
`VAULT_SKIP_VERIFY=true` is acceptable for the lab self-signed route certificate. For production, trust the cluster or cert-manager CA instead.
====

If `vault` is not installed on the bastion, run commands through the Vault server pod:

[source,bash,subs="+macros,+attributes",role=execute]
----
export VAULT_POD="$(oc get pod -n vault -l app.kubernetes.io/name=vault,component=server -o jsonpath='{.items[0].metadata.name}')"
alias vault='oc exec -n vault "${VAULT_POD}" -- env VAULT_ADDR=http://127.0.0.1:8200 VAULT_TOKEN=root vault'
vault status
----

=== Step 2 — Enable KV v2 and store the application secret

. Enable the KV v2 secrets engine at path `secret` (skip if already enabled).
. Write a secret the demo application will consume.
+
[source,bash,subs="+macros,+attributes",role=execute]
----
vault secrets enable -path=secret kv-v2 2>/dev/null || true
vault kv put secret/demo-app/config \
  api_key="workshop-api-key" \
  db_password="workshop-db-password"
vault kv get secret/demo-app/config
----

=== Step 3 — Enable Kubernetes authentication

Vault must trust tokens from the OpenShift API server. Configure the Kubernetes auth method from inside the cluster (using the Vault pod's service account as the token reviewer).

[source,bash,subs="+macros,+attributes",role=execute]
----
vault auth enable kubernetes 2>/dev/null || true
oc exec -n vault "${VAULT_POD:-$(oc get pod -n vault -l app.kubernetes.io/name=vault,component=server -o jsonpath='{.items[0].metadata.name}')}" -- \
  sh -ec 'vault write auth/kubernetes/config \
    token_reviewer_jwt="$(cat /var/run/secrets/kubernetes.io/serviceaccount/token)" \
    kubernetes_ca_cert=@/var/run/secrets/kubernetes.io/serviceaccount/ca.crt \
    kubernetes_host="https://kubernetes.default.svc:443"'
----

=== Step 4 — Create a policy and Kubernetes auth role

. Define a policy that allows **read** access to the demo secret only.
. Bind that policy to the `demo-app` service account in namespace `demo-vault-app`.
+
[source,bash,subs="+macros,+attributes",role=execute]
----
vault policy write demo-app - <<'EOF'
path "secret/data/demo-app/*" {
  capabilities = ["read"]
}
EOF

vault write auth/kubernetes/role/demo-app \
  bound_service_account_names=demo-app \
  bound_service_account_namespaces=demo-vault-app \
  policies=demo-app \
  ttl=1h
----

=== Step 5 — Deploy the sample application

. Apply the workshop manifest (namespace, service account, deployment with injector annotations).
+
[source,bash,subs="+macros,+attributes",role=execute]
----
oc apply -f setup-scripts/demo-vault-app.yaml
oc get pods -n demo-vault-app -w
----

Press `Ctrl+C` when the pod is `2/2` or `3/3` Ready (app + Vault agent init/sidecar containers).

.Injector annotations (reference)

The deployment sets annotations documented in the Agent Injector guide: enable injection, set the Kubernetes auth `role`, point at the Vault KV path, and render a small template file under `/vault/secrets/config`.

=== Step 6 — Verify the secret in the pod

. Confirm the Vault Agent wrote the secret file.
. Read it from the application container.
+
[source,bash,subs="+macros,+attributes",role=execute]
----
POD=$(oc get pod -n demo-vault-app -l app=demo-app -o jsonpath='{.items[0].metadata.name}')
oc exec -n demo-vault-app "${POD}" -c app -- cat /vault/secrets/config
----

You should see `API_KEY` and `DB_PASSWORD` values matching what you stored in Step 2.

. Optional: inspect injector logs if the secret file is missing.
+
[source,bash,subs="+macros,+attributes",role=execute]
----
oc logs -n demo-vault-app "${POD}" -c vault-agent
oc describe pod -n demo-vault-app "${POD}"
----

=== Step 7 — Clean up the demo application (optional)

[source,bash,subs="+macros,+attributes",role=execute]
----
oc delete -f setup-scripts/demo-vault-app.yaml
----

=== What you learned

* **Vault** holds the secret; the **injector** delivers it only to authorized pods.
* **Kubernetes auth** ties access to a namespace and service account instead of sharing the root token.
* Applications consume files under `/vault/secrets/` (or templates you define) without storing credentials in Git or image layers.
